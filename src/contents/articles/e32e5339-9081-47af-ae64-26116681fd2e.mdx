## なぜ Web アプリ開発から SRE に転向したのか？

僕はもともと Web アプリケーションの開発が好きで、大学時代に独学で勉強を始め、インターンや業務委託として開発の現場に携わってきました。

使用していた技術スタックは以下のようなものでした。

> **フロントエンド**

- React
- Next.js
- TypeScript

> **バックエンド**

- Go（Echo）
- TypeScript（Prisma、Ent）

2 年ほどアプリケーションエンジニアとして働く中で、徐々にモヤモヤを感じるようになりました。

> _「これ、作って本当に価値あるのかな？」_  
> _「仕様通り作っているけど、これって本当にユーザーに届いてるの？」_

そんな疑問が心に残るようになったのです。

今までに色々な機能を作ってきました。

- 管理画面からの納品書出力機能
- Redis を使った入力内容一時保存機能
- フォルダ機能
- 新規権限アカウント実装
- etc

どれもこれも、マージされたけど実際に使っている人の声を全く聞けず、自分の存在価値を問うようになってしまいました。

一方で、業務の中でふと「使われていない GraphQL スキーマを自動で削除するスクリプト」を書いてみたことがありました。開発者体験の向上を目的に、軽い気持ちで改善したものです。

それが社内の Slack で思わぬ反響を呼びました。

> _「これ改善されたの便利すぎる！」_  
> _「めっちゃ助かりました、ありがとう！」_

自分のアウトプットが目に見える形で周囲の助けになったこと、そしてその価値がすぐにフィードバックされたことに大きなやりがいを感じました。

この体験から「開発者の生産性を支える仕事」や「裏側から価値を支える仕事」に強く惹かれるようになりました。  
そして、DevOps や SRE という分野に自然と関心が移っていきました。

---

## インターンで感じた「SRE」のリアル

大学 3 年生の夏、ZOZO と freee の SRE インターンに参加し、SRE の実務を経験しました。

アプリ開発のように「新しい機能をつくる」ことが中心ではありませんが、  
**目の前に明確な課題があって、それを解決したときのインパクトが大きい**。  
この感覚が、とても自分にフィットしました。

---

## SRE の魅力：僕が惹かれた 5 つの理由

### 1. 解くべき課題が明確で、与えるインパクトが大きい

アプリ開発の仕事は、ビジネス側の仮説に基づいて機能をつくることが多いです。  
その機能が本当に使われるか、価値があるかはリリースするまで分かりません。

一方で SRE は、**すでに存在する課題（遅延、スケーラビリティ、障害、信頼性）に対して取り組む**仕事です。

- ユーザー数の増加によるシステム負荷
- 高トラフィック時のレスポンス低下
- デプロイの失敗によるサービス停止

こうした問題の解決は、直接的にサービスの安定性や信頼性に貢献できます。  
**成果が見えやすく、周囲のエンジニアの生産性にも大きく寄与できるのが魅力です。**

---

### 2. 参入障壁が高く、技術者としての市場価値を高められる

SRE は扱う技術領域が広く、かつ専門性が求められる分野です。

たとえば…

- Kubernetes / Docker などのコンテナ技術
- AWS / GCP / Azure などのクラウドインフラ
- ネットワークの知識（DNS、ロードバランサー）
- CI/CD パイプライン
- Terraform や Pulumi などの IaC
- 可観測性（Observability）やアラート設計

これらは書籍やチュートリアルだけでは習得が難しく、**「実際に運用する現場」で学ぶことで真価を発揮する**領域です。

だからこそ、経験を積めば積むほど希少価値が高まり、  
将来的には社内の技術リーダーや、社外での SRE コンサルなど、キャリアの幅が広がっていくと感じています。

---

### 3. 個人では経験できない「仕事でしかできないこと」がある

正直、Web アプリ開発は個人でもある程度のレベルまで経験できます。  
一人で ToDo アプリやブログを作って、デプロイまでやった人も多いはず。

でも、SRE 領域で扱う「大規模サービスのトラフィック制御」や「本番障害対応」などは、  
**個人開発では絶対に経験できない現場ならではの課題**です。

こうした経験は技術者としての引き出しを増やし、今後のキャリア選択に大きく役立つと実感しています。

---

### 4. 学んだ技術が陳腐化しにくく、長く使える

フロントエンド領域では、数ヶ月単位でトレンドが変わることもあります。  
React、Vue、Next.js、Remix、Server Components… 学習サイクルがとても早い世界です。

対して、SRE が扱う技術（インフラ、ネットワーク、監視基盤、セキュリティなど）は、  
**一度深く理解すれば数年は通用する「原理や設計思想」に関わるものが多い**です。

学習コストは高めですが、その分得られる知識やスキルは長持ちする。  
この「学びが資産になる感覚」も自分にとって魅力的でした。

## まとめ：SRE は“価値に近いところ”で働ける仕事

SRE という仕事は、ユーザーの目には直接触れないかもしれない。  
でも、**サービスの安定性、生産性、そして信頼性を支える「土台」をつくる**仕事です。

僕が SRE に転向した理由は、単なる技術的な好奇心ではなく、  
「価値ある仕事をしていると感じたい」「人の役に立っている実感を得たい」  
という気持ちに根ざしたものでした。

もし、今あなたが「アプリ開発だけでいいのか？」と感じているなら、  
一度 SRE という視点からエンジニアリングを見直してみるのも面白いかもしれません。
